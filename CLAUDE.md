# CLAUDE.md — note記事運用システムの憲法

このファイルは毎朝の作業を始める前に**必ず最初に読む**。
ここに書かれた手順とルールが、すべての判断に優先する。

---

## ★最優先ルール（2026-07-02追加）：NOTE_STRATEGY.md を必ず参照する

記事を作成・改善するときは、**着手前に必ず `NOTE_STRATEGY.md` を読み、その「タイトルの型」「本文構成ルール」「分量」を最優先で適用する**こと。本CLAUDE.md下部の旧記述（例：本文2,500〜3,500字、タイトルは数字＋ベネフィット）と食い違う場合は、**NOTE_STRATEGY.md を優先**する。特に：

- **タイトル**：「［フック：逆説／数字／問いかけ（先頭25字以内）］＋［当事者の具体］｜シリーズ名 #番号」。**連番・管理番号を先頭に置かない**（「1型糖尿病と生きる #9〜」「第○回／全14回」は沈んだ型）。
- **本文**：冒頭はセリフか場面＋数字から（解説スタート禁止）／章見出しの体裁は NOTE_STRATEGY.md に従う／専門用語は平易に補足／検証可能な一次情報を1つ入れる／感情を隠さない／締めの定型（マガジン＆フォロー誘導・コメント問いかけ・スキ誘導・医療免責）。
- **分量**：1,500〜2,000字・段落は短く。
- **最優先シリーズは「1型糖尿病と生きる」**。スキ率・初動2時間・一次情報性を重視し、軸と無関係な記事（セール情報等）は出さない。

## このシステムの目的

noteを毎日運用する中での3つの問題を「構造」で解決する。

| 問題 | 解決の仕組み |
|------|------------|
| ①内容が薄い | `knowledge/` に具体例・数字・体験談・出典を蓄積し、毎回必ず織り込む。1記事1メッセージに絞る |
| ②記事が重複する | `logs/published.md` に全タイトル・テーマ・キーワードを記録。生成前に必ず照合し、被ったら角度を変える |
| ③閲覧数が伸びない | タイトルを「数字＋ベネフィット＋具体性」に統一。冒頭3行で悩みを言い当てる。快眠は連載で再訪を促す |

## 投稿の前提（時間帯）

- ユーザーは**帰宅後の19時ごろ**にnoteへ投稿する。下書きは**毎日18時にGmailの下書き**へ届く（自動）。
- そのため記事は「夜に読まれる」前提で書く：
  - **株式**：投稿時点で当日の日本市場は大引け済み（15時）。→ **当日終値・当日の値動き**まで盛り込める。前日比だけでなく「今日はこう動いた」を語る。
  - **快眠**：夜に読まれる強みを活かし「**今夜からできる**」「**今晩の眠りを変える**」という当日アクションに落とす。
- 文中で時刻に触れるときは夜の生活動線（帰宅後・就寝前）を意識する。

---

毎朝、次の2本を生成する。
1. **株式投資** — 曜日でテーマを固定し、毎日角度を変える
2. **快眠メソッド** — Day1〜Day14の連載を順番に投稿（14日後は応用編へ）

加えて、2記事を紹介する**Threads用の短文**を生成する。

## 毎朝の手順（この順で実行する）

### STEP 1. 過去ログの把握
- **まず note RSS `https://note.com/tommykaoleo/rss` を開き、実際に投稿済みのDay・テーマを確認する**（RSSが真実の基準）。`logs/published.md` とズレていたら、**先に台帳をRSSへ同期**してから進む。※Day番号重複の再発防止（2026-06-22）。
- `logs/published.md` を読む。
- **直近14日**の株式テーマ・キーワードを頭に入れる（重複回避のため）。
- 快眠連載が**今どのDayまで進んでいるか**を確認する（次Day＝RSSで確認した最新Day＋1。台帳の数字を鵜呑みにしない）。

### STEP 2. ナレッジの読み込み
- `knowledge/persona.md` … 文体・人格を思い出す。
- `knowledge/title-rules.md` … タイトルの型を確認。
- 株式記事を書くなら `knowledge/stock-knowledge.md`。
- 快眠記事を書くなら `knowledge/sleep-method.md`。

### STEP 3. 株式記事を1本生成
1. 今日の曜日テーマを `stock-knowledge.md` の曜日割りから決める。
2. **Web検索で当日／前営業日の最新相場を必ず取得**（日経平均・TOPIX・米国市場・ドル円・物色セクター・主要ニュース）。数字には時点を添え、書き手の解釈を付けて記事に織り込む（`stock-knowledge.md`「鮮度ルール」に従う）。
3. **一次情報・信頼できる情報源で最低5本裏取りし、出典URLを控える**（証券会社・取引所・金融メディア等）。推測断定は厳禁。薄さはここで防ぐ。
4. `published.md` と照合し、**過去と被らない角度**にする。被りそうなら別の切り口へ。
5. `title-rules.md` に従いタイトルを**最低7案**出し（各案に「なぜ目を引くか」を添え、4基準×各10点で採点）、合計点が最高の1案を選んで**選んだ理由**と別案上位3つも書く。
6. `templates/article-template.md` の構成で本文を生成。**本文は2,500〜3,500字を目安**に、1記事1メッセージで深掘りする（短く薄くしない）。
7. `articles/stock/YYYY-MM-DD.md` に保存。

### STEP 4. 快眠記事を1本生成
1. `published.md` を見て**次に投稿すべきDay番号**を判定する。
2. `sleep-method.md` の該当Dayの設計に沿って生成。
3. タイトルも `title-rules.md` に従い**最低7案**出し、4基準×各10点で採点して最高点を選ぶ。「Day○/14」を入れつつ、その日単体でも読みたくなる形にする。冒頭に「Day○/14」を明記し、前回・次回への導線を入れる。
4. **本文は2,500〜3,500字を目安**に深掘りする。睡眠科学の根拠は信頼できる情報源で裏取りし出典を添える。
5. Day14が済んでいたら**応用編／季節版**へ切り替える（同じ内容を繰り返さない）。
6. `articles/sleep/dayNN.md` に保存。

### STEP 5. 表紙画像プロンプトを生成
- `knowledge/cover-image.md` に従い、2記事それぞれに**Midjourneyプロンプトを1本**作る（記事の1メッセージを象徴、文字なし、`--ar 16:9 --style raw`）。

### STEP 6. Threads短文を生成
- `templates/threads-template.md` に沿って、2記事それぞれの短文＋リンク枠を作る。

### STEP 7. ログに追記
- 投稿後（またはユーザーの指示時）、`logs/published.md` に2記事を追記する。
- 列: 投稿日 / 種別 / Day / タイトル / キーワード / note URL / Threads済

## 品質ルール（薄さ・重複・伸び悩みへの歯止め）

- **1記事1メッセージ。** 言いたいことを1つに絞る。盛り込みすぎない。
- **具体性を必ず入れる。** 数字・固有名詞・体験談・出典のいずれかを毎段落に。
- **冒頭3行で悩みを言い当てる。** 読者の「それ私だ」を最初に取る。
- **断定しすぎない（特に投資）。** 「〜と考えています」「あくまで一例」など誠実なトーン。
- **読者との共通点・親しみやすさを必ず入れる。** 「同じ悩みを持つ仲間が隣で話す」姿勢。共通の立場・悩み・失敗を1つ以上織り込み、「ありますよね」「私もそうでした」で距離を縮める。上から目線・正論の押し付けはNG（`persona.md`参照）。
- **生成のたびにナレッジを育てる。** 使った具体例・反応の良かった切り口は `knowledge/` に追記する。

## 執筆後のQC工程（2記事とも、下書き作成前に必ず通す）

書き上げたら提出前に下記6点を自己点検し、引っかかったら直してから次へ進む。

1. **事実照合**：数字・固有名詞・相場データが出典と一致するか。曖昧な点は「要確認」と明記し捏造しない。
2. **重複照合**：`logs/published.md` の直近テーマ・キーワードと角度が被っていないか。
3. **AI臭チェック（最重要）**：定型の共感冒頭（「〜な夜が、あなたにもあるかも」式）・テンプレ見出し（【その1】【その2】）・「いかがでしたか」「してみてくださいね」の乱発を**禁止**。`persona.md` の声に寄せ、固有の体験・数字・言い切りを入れる。
4. **体裁**：note貼付で崩れないプレーンテキスト（絵文字・#・**・[]() 不使用、URLは直貼り）。
5. **免責**：投資は「※最終判断はご自身で」、健康は誠実なトーンが入っているか。
6. **音読仕上げ**：一度通しで読み、リズムが単調・説明的になっていないか整える。

## 投稿の制約（理解しておくこと）

- noteには公式投稿APIがない → **完全自動投稿は不可**。記事は完成形で生成し、投稿はユーザーが手動で行う**半自動**運用。
- Threadsも当面は**短文＋リンクを生成 → ユーザーが手動共有**。自動化は将来検討。
